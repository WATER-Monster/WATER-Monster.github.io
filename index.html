哈希表相关

Posted on 2020-06-30 | In 算法基础
    给定两个数组，编写一个函数来计算它们的交集。
    这种题目乍一看感觉可以 for循环 更小的那一个数组，用 if in 来判断元素是否在两个数组中都存在，但这样做时间复杂度是O(n²)级别，是最慢的那种解决方式。
    之前跟一位面试官聊过这个问题，他提出可以先对两个数组做排序，然后用两个指针遍历两个数组，谁的元素小谁多跑一位，元素相等就一起跑一位。这方法挺好，复杂度是快速排序 + O(n+m)，两个数组都跑完就可以拿到结果。不过这个方法只适用于可以做排序的数组，如果两个数组内部都有无法排序的元素就行不通了。
    最好的解决方法是 HashMap ，用 哈希表 记录较小的数组内每个元素出现的次数，存成kv，遍历较大的数组，如果元素存在于哈希表内且次数为正数，那么将元素存入结果集，次数减一即可。这样做时间复杂度是O(n+m)。

jwt rsa 密钥

Posted on 2020-06-20 | In jwt rsa
    我们普通业务流程下使用 token 对用户的登陆进行验证，jwt 内部也有非常简单便利的 jwt 生成 token 的方法。但直接传递 token，在某些业务背景下是非常不安全的，因为会被第三方直接截取，得到你的 token。所以在安全性需求高的接口中，使用 token 进行直接校验是不可取的。
    那么这种情况下，我们就需要用到非对称加密了，其实已经到了用非对称加密的程度的话，直接用 https 能解决很多问题。不过，ssl证书是要钱的嘛，我们只希望保证会话安全的情况下，还是自己做一步非对称加密好了。jwt 内部为我们提供了 rsa 加密的算法，即非对称加密的算法，使用起来也很简单：
    这里简单讲下非对称加密的原理，非对称加密算法需要两个密钥：公开密钥（publickey）和私有密钥（privatekey）。公开密钥与私有密钥是一对，如果用公开密钥对数据进行加密，只有用对应的私有密钥才能解密；如果用私有密钥对数据进行加密，那么只有用对应的公开密钥才能解密。我们手上有一对密钥，在向客户端发送数据时发送公钥，客户端的数据使用公钥加密，传输给服务器端，服务器端使用私钥解密。私钥只有我们自己知道，传输过程即使被劫持，劫持者没有私钥，也无法解密请求的内容。
    很多人对非对称加密一知半解，以为它的实现是依靠代码设计上实现用双重密钥进行加解密的，但其实非对称加密的原理与计算机科学没有一丝一毫的瓜葛，它是纯数学领域实现的。有兴趣的朋友可以参考这篇文章 https://www.jianshu.com/p/ad3d1dea63af ，这位作者对它的原理讲解的很细致。
    如果想要自己做非对称加密，就需要自己使用 openssl 开源库来生成一对自己的公钥私钥，是的，上面的那对公钥和私钥可不是随便乱填一下就行了的。openssl 的安装与使用这里也不再多讲，按照网上的流程一步一步来就可以了。

mysql 索引

Posted on 2020-06-17 | In mysql 相关
    我们都知道对索引的优化是 MySQL 优化中相当重要的一环，但很多人对索引的理解其实都很浅，认为它就是一个目标数据的导航，认为无论什么样的查询速度慢，只要加个索引就能解决。我们今天就来讨论下，MySQL 索引优化相关的一些重要的点。
    1.索引的缺点，为什么不能盲目的建立索引。
     索引可以大幅提高我们针对索引字段的查找效率，但我们要知道，MySQL 在插入和更新以及删除数据时，同样也需要维护表内的索引。如果你的索引过多，一些不频繁查询的字段也加了索引，就会在插入和更新的时候吃苦头。并且索引其实也是张表，也要占磁盘空间，大量的数据配上大量的索引会急剧增大磁盘的占用量。
    2.表之间 join 的时候怎么建立索引？
    如果我们使用 jion 查询表关联数据，而关联字段在左表和右表内都未建立索引，那么我们需要怎样为该字段建立索引，给左表的字段建立？还是给右表的字段建立？或者两个表都建立一个该字段的索引？我们知道 left join 是左表的全有数据加右表的匹配数据，right join 则相反，是右表的全有数据加左表的匹配数据。所以我们给 join 查询建立索引的时候，左查询需要给右表字段建立索引，右查询给左表字段建立索引，在 join 查询里给左表或者右表的全有数据加索引是毫无意义的，因为这些全有数据最终都需要提取出来，而另一表的匹配数据如果有索引则能更快定位。
    3.联合索引怎么建立，联合索引怎么使用？
    在实际业务场景中我们经常会遇到一条查询语句需要对多个字段做筛选的情况，而MySQL在一次查询中只能使用一个索引。这时候我们对单个字段建立的索引就无法覆盖到我们业务中涉及到的所有字段，这条查询语句要优化，就需要建立联合索引。联合索引的建立方式其实很简单，create index on table(col1,col2,col3),即可。联合索引的使用需要遵守一个很重要的原则：查询从索引的最左前列开始且不能跳过索引中的列。这是什么意思呢？我们的语句按 create index on table(col1,col2,col3) 建立的联合索引，在 where 后跟 col1 可以走到索引，跟 col1，col2也可以走到索引，跟 col2， col3 不能走到索引，跟 col1，col3 只能走到 col1 的部分索引。联合索引怎么建立的，就要按怎么样的字段顺序来，从左至右，中间的字段断了缺了都只能走到部分索引，即左边开始按顺序来的字段的有效索引，从断掉的部分开始，后续字段的索引一概用不到。
    4.不要在索引的字段上做操作。
    这句话是什么意思呢，假设我的 col1 是有索引的字段，但我的 sql 语句是这么写的：select col1 from table where LCASE(col1) = ‘xxx’。这样的 sql 语句的 col1 列的索引是无效的，MySQL这时会走全表扫描。
    5.少使用 ‘> < !=’ 这类的操作符。
    涉及到范围查询的时候，单个字段的索引通常不会有太大问题，但是对于联合索引，即便你遵循了联合索引的最左原则（就是上面讲的），遇到比较类操作符的时候，做比较类操作的字段的后续字段的索引会全部失效。仍然以上面我们的联合索引 create index on table(col1,col2,col3) 为例，如果我们的 sql 后段是 where col1 = ‘xxx’ and col2 > ‘xxx’ and col3 = ‘xxx’，那么从 col3 开始，包括 col3 本身，他们的索引是用不到的。在使用联合索引的时候，一定要留意你的范围查找操作，必要的时候可以置换联合索引顺序，将你必须要使用范围查询的字段放在联合索引的最后。
    6.like通配符以%开头无法走到索引。什么是覆盖索引。
    这个其实绝大多数人都知道，模糊匹配如果直接以 % 开头匹配是走不到索引的。不过其实有一个解决方案，就是使用覆盖索引。什么叫覆盖索引？就是指一个查询语句的执行只用从索引中就能够取得，不必从数据表中读取，比如 select col1， col2 from table where col1 = xxx and col2 like ‘%xx%’，如果我们像上述那样有col1， col2， col3 的联合索引在，那么这条语句就可以走到覆盖索引，即便我们的 col2 模糊查询是从 % 开始匹配的，它的查询也能被我们的联合索引所覆盖。
    索引的优化其实还远不止这些，我会慢慢的在以后的博客里讨论到。

kafka group

Posted on 2020-06-16 | In kafka 相关

kafka 相关

    之前我们聊了聊消息队列是做什么用的，然后聊了下 kafka 是个什么东西，那今天我们就来详细的聊下，kafka 的 group 是用来做什么的。
    我们前面说到，kafka 的一个 topic 可以配置多个 partition，不同的 partition 可以放在不同的 kafka server 上，这样的机制保证了 kafka 可以很方便的做分布式化，横向扩展。那么我的一个 consumer 消费者，从不同的 partition 上，就可以接收到 topic 内的全部消息，但现在有一个问题，如果我有三个 consumer 实例，我会消费到多少条消息呢？topic 内消息总数乘以 3 吗？显然不会，我的三个实例应该是同一个处理逻辑，那不是同一个消息被我处理了三次？所以真实情况是我的三个实例总共消费了一整份 topic 内的消息，而不应该是三整份。
    现在又有一个问题来了，如果我的另一组实例，执行不同的逻辑，但也需要消费这一份 topic 内的消息，该怎么办，我这 topic 已经被前面的三个实例消费过了啊，放在其他的简易消息队列里，比如 redis 的消息队列，我消费完就该把 topic 内消费掉的消息删了。
    kafka 的 group 就是解决上面这两个问题被设计出来的。在你指定你的头三个实例在同一个 group 下的时候，你的三个 consumer 是共同消费一整份消息的，不会出现我三个同样的 consumer 实例把同样的事情做了三次的情况。而你把后面那组实例放在另一个 group 里后，第二组实例的 consumer 也同样能消费掉一整份 topic 消息。
    这样 group 的作用就很明确了，如果说 topic 是消息的分组的话，group 就是你不同功能的 consumer 的分组。一组人干一件事，另一组人也可以用同样的 topic 干另一件事，双方消费消息是互不干涉的。
