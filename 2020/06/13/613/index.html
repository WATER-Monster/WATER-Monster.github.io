为什么 列表推导式 会比普通的 for循环 更快？

我们都知道，python 的 for循环 的效率其实是相当慢的，在使用for生成一个列表时，使用列表推导式会比直接写一个for快很多，但是这是为什么呢？
    我们先来介绍下 python 的一个模块，dis模块。dis 模块可以对你的 python 程序进行反汇编，生成字节码指令，我们可以使用这个模块，我们可以写一个简单的函数，然后使用 dis模块来看一看一个python函数在被执行时到底做了哪些事。
    这是 func1 在第七行走for循环时的字节码指令。
    这是 func2 在第十四行列表推导式时的字节码指令。
    我惊讶的发现，3.6的python在走列表推导式的时候甚至没有做 FOR_ITER ，即for循环迭代。我在这之前看到有人说过，‘python 在列表推导中直接使用了‘LIST_APPEND’这个字节码来实现 append 功能，效率相当的高。而在 for 循环中每次循环都要先载入 append 这个属性然后再 ‘CALL_FUNCTION’一下。这样势必就会慢了很多。’ 但他测试的版本应该是基于python2，现在看来在 python3 中列表推导式连自己 iter 一遍都不需要了，直接用源列表的 generater 生成自己的新列表。
    这个结论目前还有待推敲，但我们可以知道，python3 在 python2 的基础上使用生成器改变了for循环确实带来了很大的变革。列表推导式的生成方式比直接写for循环的效率要高出很多，这恐怕不仅仅是 LIST_APPEND 这个层面上的区别，更是 python3 for循环遍历使用 generater 即生成器带来的优化。
